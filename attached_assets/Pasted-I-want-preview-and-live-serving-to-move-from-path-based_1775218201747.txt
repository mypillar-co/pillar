I want preview and live serving to move from path-based URLs to subdomain-based URLs.

Current pattern:
mypillar.co/sitename

Target pattern:
organization.mypillar.co

This should apply to both preview and live serving.

GOAL

Each organization/site should feel like its own real website with its own host, not a page inside the main app.

Examples:
norwinrotary.mypillar.co
irwinmarket.mypillar.co
shidlelodge.mypillar.co

And for preview:
preview-norwinrotary.mypillar.co
or
norwinrotary.preview.mypillar.co

Pick one preview convention and use it consistently.

REQUIREMENTS

1. Host-based routing
Add host-based site resolution so the server can determine the site from the request hostname, not only from the URL path.

Examples:
norwinrotary.mypillar.co → resolve slug/site from subdomain
preview-norwinrotary.mypillar.co → resolve preview site
custom domains can come later, but architect for it now

2. Wildcard subdomain support
Set up the app/server to support wildcard subdomains for *.mypillar.co.
The routing layer must parse the hostname and map the leftmost subdomain to the correct site record.

3. Site resolution rules
Implement a site resolution helper like:
resolveSiteFromHost(hostname)

Rules:
- strip port if present
- normalize case
- support apex/root domain separately
- support www safely
- support preview hostnames separately from live hostnames
- reject invalid hostnames cleanly

4. Suggested hostname mapping
Live:
<site-slug>.mypillar.co

Preview:
preview-<site-slug>.mypillar.co
or
<site-slug>.preview.mypillar.co

Choose the one that is easiest to implement cleanly.
I prefer something simple and obvious.

5. Database support
Ensure each site has a unique slug that is safe for subdomain use.
Add or confirm:
- normalized slug
- uniqueness enforcement
- reserved subdomain handling

Reserved examples:
www
app
api
preview
admin
mail
docs

Do not allow site slugs that collide with reserved infrastructure subdomains.

6. Preview/live separation
Preview should still be isolated from live.

Rules:
- live subdomain serves published output
- preview subdomain serves draft/compiled preview output
- preview should require authenticated access or signed access
- live should remain public

7. No more /sites/:slug as the primary UX
You can preserve old path-based routes for backward compatibility internally if needed, but the primary user-facing preview/live experience should be host-based subdomains.

8. Serving behavior
The public site renderer should work exactly the same as now, but site lookup should happen from the hostname.

Meaning:
- request comes in for norwinrotary.mypillar.co
- server resolves site by subdomain
- serves the compiled HTML for that site

9. Preview behavior
Preview pane in dashboard should load the actual preview subdomain in an isolated iframe/webview.

Example:
https://preview-norwinrotary.mypillar.co

This should behave like a real browser visit to that site, not a dashboard reconstruction.

10. Local/dev support
Implement a clean development fallback.
Examples:
- continue supporting path-based local preview in dev
- or support host header mapping in dev if practical

But production architecture should be subdomain-first.

11. Security / correctness
Ensure:
- hostname parsing cannot escape tenant boundaries
- preview hostnames resolve only to the correct org/site
- published hostnames serve only published content
- draft preview is not accidentally public

12. Deliver back
Tell me:
- how subdomain resolution will work
- what DNS/wildcard setup is needed
- what server/router changes are needed
- what DB/site slug constraints are needed
- how preview auth will work
- whether old path routes remain temporarily for compatibility
