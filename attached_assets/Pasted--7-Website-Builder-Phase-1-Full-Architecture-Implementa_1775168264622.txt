#7 - Website Builder Phase 1 — Full Architecture Implementation

What & Why

Turn Pillar from a single-template HTML blob generator into a structured, dynamic website operating system built for production SaaS. Seven architectural layers, mandatory org isolation, event-driven job processing, block-level render caching, full observability, versioned publish history, and explicit failure safety. The renderer never queries dashboard tables directly. AI never rewrites identity content. One block failure never breaks a site.

Architectural philosophy

These four priorities govern every implementation decision in this task. When any tradeoff arises, the executor must apply them in order.

1. Determinism over AI If a correct answer can be derived from structured data, it must be. AI is called only when structure runs out. The page planner, site profile builder, adapter layer, renderer, and compiler are all deterministic. AI touches only two things: structured contentJson payloads during initial generation, and structured ImportFinding extraction during import. It does not touch rendering, routing, schema logic, or any auto-update path.

2. Structure over generation Sites are not generated — they are assembled from structured records. A site is a site_pages record + site_blocks records + site_themes record + site_nav_items records + site_block_bindings records. generated_html is a compiled artifact of those records, not their replacement. If the structured records are lost and only generated_html remains, the site cannot be edited, updated, or maintained — that outcome must be architecturally impossible.

3. Data integrity over convenience Org isolation is not optional. Every query in the site system must carry WHERE org_id = ?. There are no global queries. No service infers orgId from context — it is passed explicitly as a required parameter. Financial data is sanitized at the adapter boundary — it never flows into the renderer. Identity content (mission, about, history, programs) is locked against AI modification after initial generation. Stats are omitted rather than estimated. A missing field produces a missing block — not a fabricated one.

4. Separation of concerns over speed The dashboard modules do not connect to the renderer. The renderer does not query the database. The adapters do not render HTML. The page planner does not generate content. The content generator does not persist records. The compiler does not make AI calls. Each layer has one job. Violating this separation — even for a small convenience — creates the coupling that causes data leaks, AI drift, and debugging nightmares at scale.

AI's role in this system: AI assists. Architecture controls. AI is a collaborator that proposes structured content payloads and extracts structured import signals. The architecture decides what gets built, how it renders, what gets updated, and what stays locked. If AI is ever asked to do something the architecture controls — write HTML, decide block order, update identity content, compute metrics — that is a bug, not a feature.

Non-negotiable principles

Structured records are the source of truth
generated_html is compiled output only — never edited directly
Import must curate and map — not scrape and rewrite
Site planning must happen before rendering
AI assists extraction, planning, and drafting of structured content — it does not write HTML or CSS
Event behavior must be explicit — not inferred ad hoc throughout the app
Auto-update operates at the block/module level only
Identity content (mission, about, history, programs) is stable — AI may only touch it during initial generation, never during auto-update
Dashboard data must never connect directly to the renderer
All website integrations flow through the site data integration layer
Every site service receives orgId as a required parameter — no global queries allowed
One block render failure must never break the full site compile
Adapters must sanitize financial data — renderer may never expose transaction IDs, payer emails, fees, refunds, or internal revenue data
Done looks like

Generated sites are composed from structured records in site_pages, site_blocks, site_nav_items, site_themes
generated_html is compiled from those records by siteCompiler.ts using the appropriate compile mode
Different org types produce different page plans (pages, section order, variants)
Events render with correct CTA language, ticket links, and sold-out states based on explicit event fields
Sponsor grid auto-updates from approved sponsors via the sponsor adapter
The renderer has no direct queries to dashboard tables
Stats are never fabricated — sections omit when data is absent
Auto-update writes to site_change_log for every change and touches only auto_apply blocks
AI may not overwrite mission, about, history, or programs during auto-update
Site import stores structured findings with user review before application
Every publish creates a site_versions record
Block render failures return <!-- block failed: [type] --> and compilation continues
All jobs are queued via job_queue — no polling logic inside services
site_system_logs captures all generation, import, render, auto-update, and adapter failures
site_render_cache prevents redundant block re-renders when data is unchanged
org_activity_stream events trigger targeted jobs instead of full scans
Out of scope for Phase 1

Drag-and-drop visual block editor UI
Full revision history UI beyond site_change_log and site_versions
Public-facing vendor/participant roster modules (adapters and bindings set up; rendering is Phase 2)
Exhaustive multi-page crawler (homepage + up to 5 detected sub-pages)
Complex visual design controls beyond theme presets
Custom domain routing changes
Approvals/board links on the public site (stays internal)
Rate limiting implementation (architecture is designed for it; enforcement is Phase 2)
Production readiness requirements (must be complete before paying customers)

These are enforced as acceptance criteria for this task:

Org isolation enforcement (resolveOrgScope middleware on all site routes)
Error logging (site_system_logs written by all services on failure)
Job queue (job_queue table; all async work enqueued, not polled inline)
Data adapters (all 8 adapters implemented, sanitized outputs only)
Site versions (created on every publish)
Block failure safety (renderer returns comment placeholder, continues)
Metrics caching (site_render_cache and event_public_metrics)
Stripe data sanitization (adapters strip all financial internals)
Audit logs (site_change_log written on every auto-update and publish)

4. Add resolveOrgScope middleware and enforce org isolation

Create artifacts/api-server/src/middleware/resolveOrgScope.ts. This middleware runs on all /api/sites/* routes and all site-adjacent routes (events, sponsors, etc. when called from site context).

It injects into req: orgId (from session), siteId (looked up from sites where orgId = session.orgId), userRole (from session). It returns 403 if orgId is missing. It returns 404 if the requested siteId does not belong to session.orgId.

Rule: Every service function in lib/site/services/ takes orgId: string as its first or second parameter. No service may query site data without an orgId filter. No query in the site system may be made without WHERE org_id = ? or equivalent Drizzle .where(eq(table.orgId, orgId)).

5. Write lib/site/services/siteLogService.ts

Wraps all writes to site_system_logs. Exports: logInfo(service, operation, message, meta?), logWarning(service, operation, message, meta?), logError(service, operation, message, meta?, error?). All service functions call these on failure. On success of major operations (generate, compile, publish), call logInfo. The metadataJson field receives the error stack and any relevant IDs. This service never throws — log failures are swallowed silently after a console.error.

6. Write lib/site/services/jobQueueService.ts

Exports:

enqueueJob(jobType: JobType, orgId: string, payload: JobPayload, scheduledAt?: Date): Promise<string> — inserts into job_queue, returns jobId

The scheduler (siteSchedules) calls processJobs() on its interval instead of calling services directly.

18. Write lib/site/services/siteCompiler.ts

Function: compileSite(orgId: string, siteId: string, mode: CompileMode, db): Promise<string>

CompileMode: 'full_compile' | 'block_compile' | 'page_compile'

Mode rules:

full_compile: triggered by generation, publish, theme change — renders all blocks from scratch, ignores cache
block_compile: triggered by event/sponsor/announcement updates — renders only blocks whose dataHash has changed per site_render_cache; all other blocks read from cache
page_compile: triggered by manual block edits — renders only blocks on the affected page
Full compile flow:

Assert orgId matches sites.orgId — reject if not (isolation check)
Fetch all site records scoped to orgId
Call siteDataIntegrationService.getSiteData(orgId, siteId, allBindings, db) → SiteDataMap
Render nav HTML from header nav items
For each visible page, for each visible block:
Compute dataHash from block + its live data slice
On full_compile: skip cache, render directly
On block_compile/page_compile: check site_render_cache — return cached HTML if hash matches, else render and update cache
Call blockRenderer.renderBlock(block, theme, liveData) — on failure it returns a comment placeholder and logs the error; compilation continues
Render footer HTML from footer nav items + org contact info
Wrap in CSS custom properties from site_themes
Write generatedHtml and compiledAt to sites
Write site_system_logs entry (info on success, error on failure)
Return compiled HTML
