PART 2 — MULTI-USER ORGANIZATION FOUNDATION

This is the most important architectural change.

The current product feels too much like one user owns one organization.
I need you to move the codebase toward a real multi-tenant organization model.

Add the foundation for:
- organization_members table
- roles such as owner, admin, finance, events, content, viewer
- membership-based organization access instead of owner-only access
- invitation-ready architecture, even if full invite flows are not completed in this pass
- reusable role / permission guard utilities

Wherever access currently assumes:
organization.userId === req.user.id

Refactor toward:
user is a member of organization with role-based permissions

Do not break the existing single-owner setup.
If needed, preserve backward compatibility while introducing the new membership model.

PART 3 — SECURITY HARDENING

Review the current authentication, session, storage, and API handling and fix obvious weaknesses.

Specifically review and improve:

1. Object / document access
   - Ensure private uploaded documents are never retrievable without proper auth and org-level authorization
   - Enforce storage ACL checks consistently on download/read routes
   - Confirm presigned upload flow is scoped safely

2. CSRF / request protection
   - Add practical protection for state-changing authenticated routes
   - At minimum validate Origin / Referer or implement CSRF token strategy where appropriate

3. Session handling
   - Review session rotation and security posture
   - Keep secure cookie handling strong

4. XSS risk
   - Review any stored HTML / embed code flows
   - Strengthen sanitization where needed
   - Reconcile any conflict between allowed embed behavior and CSP

5. Stripe webhook handling
   - Confirm webhook signature verification is correct
   - Confirm idempotent processing of important events
   - Make webhook behavior production-safe

When making changes, prefer targeted hardening over broad rewrites.
