I want to add an important requirement to the website generator and site import system.

The generator can access an organization’s current website and extract content/images, but right now the imported results become messy, generic, and structurally weak.

I need the importer and auto-update logic to become much more disciplined.

IMPORTANT PRODUCT GOAL

If I provide an existing website URL, Pillar should use that site as intelligently as possible.

That means:
- preserve useful content
- preserve useful structure
- preserve brand cues
- preserve real images/logos where possible
- avoid flattening everything into generic AI copy
- avoid importing junk, repetition, or broken layout logic

The existing site should be treated as:
1. a content source
2. a brand/reference source
3. a structural clue source
4. an image/media source

Not as a blob of text to paraphrase loosely.

PART 1 — IMPORT STRATEGY

Redesign the site import pipeline so it does NOT just scrape text and summarize it.

Instead, implement a structured import process:

Stage A: Crawl and classify
When importing an existing site, identify:
- homepage
- about page
- programs/services pages
- events pages
- contact page
- join/get involved page
- sponsor/partner pages
- donation pages
- blog/news/announcement pages if present

Do not treat all pages equally.
Classify page purpose before extracting content.

Stage B: Extract structured signals
From the current site, extract and normalize:
- organization name
- tagline
- mission statement
- about text
- program/service descriptions
- recurring event information
- contact info
- location / meeting time
- social links
- CTA patterns
- page titles / navigation structure
- logos
- image assets
- colors / visual cues where useful
- sponsor or partner references
- board/member info if present

Stage C: Quality filtering
Do NOT import everything.
The importer should detect and avoid:
- repetitive boilerplate
- navigation junk
- cookie/privacy text
- footer clutter
- duplicated phrases
- stale announcements
- broken formatting
- low-quality decorative images
- irrelevant stock imagery
- empty sections

Stage D: Rebuild from structured content
The imported site should be rebuilt into the new Pillar modular system.
Do not try to mimic the source site by pasting its content into one generic layout.
Instead:
- extract the best content
- map it into a clean site spec
- generate a better modern site using the imported content as source material

PART 2 — IMAGE HANDLING

Image import must be smarter.

When a site is imported:
- prioritize the organization’s real logo first
- preserve real photos from the site if they are relevant and good quality
- identify hero-worthy images vs supporting images
- avoid tiny, low-resolution, decorative, or duplicate images
- avoid cluttering the generated site with too many weak images

Create image selection logic that ranks images by:
- relevance
- resolution
- uniqueness
- likely importance on original page
- whether it appears to be a logo, banner, event photo, sponsor logo, etc.

If no good real images exist, gracefully fall back to placeholders or curated stock imagery.
But do not mix random imported junk with generic stock in a way that looks messy.

PART 3 — CONTENT PRESERVATION RULES

If the original site has strong, specific copy, preserve it as much as possible.
Do NOT rewrite everything into generic AI nonprofit language.

Examples:
- preserve a strong mission statement if it exists
- preserve named programs/projects if they exist
- preserve real location, meeting schedule, and contact details
- preserve actual calls to action like Join Us / Volunteer / Become a Member
- preserve organization-specific language and tone where appropriate

AI should improve clarity and structure, but not erase identity.

The rule should be:
enhance and reorganize strong source material
not replace it with generic filler

PART 4 — PAGE PLANNING USING EXISTING SITE

When a source site exists, page planning should use it as a strong input.

Examples:
- if the current site clearly has About, Programs, Events, Contact, use that structure as a starting point
- if it has a strong membership/join flow, preserve that concept
- if it is highly event-driven, prioritize event-driven architecture
- if it has sponsor/partner visibility, keep that
- if it has recurring announcements/news, recognize that pattern

Do not flatten every imported site into the same page structure.

PART 5 — AUTO-UPDATE LOGIC

If a user enables auto-update, I want the logic to be sound and conservative.

Auto-update should NOT mean:
“AI freely rewrites the website whenever it sees something.”

That is dangerous and will create drift and low trust.

Instead, define content types by update behavior.

A. Safe to auto-update automatically
These can update without major risk:
- upcoming events lists
- ticket availability / event status
- sponsor lists tied to approved sponsor data
- announcement feeds
- board-approved dynamic notices
- basic contact changes from structured settings
- social proof counters if sourced from real platform data

B. Safe to suggest but not auto-publish
These should generate proposed edits for review:
- homepage headline changes
- mission/about rewrites
- new CTA wording
- featured event highlights
- homepage promo blocks
- seasonal campaign banners
- AI-written summaries of recent activity

C. Never auto-rewrite without approval
These should require explicit review:
- organization history
- core identity language
- programs/services descriptions
- board/member information
- legal/policy text
- donation language
- partner/sponsor claims
- major visual/layout changes

PART 6 — CHANGE DETECTION / AUTO-UPDATE RULES

If auto-update is enabled, use sound logic:

1. Prefer structured platform data over scraped inference
If Pillar has event records, sponsor records, announcements, or settings in its own database, those should be the source of truth.

2. Do not rewrite whole pages for a small data change
If only an event list changed, update the event module only.

3. Scope updates to modules
Auto-updates should operate at the section/module level, not regenerate the entire site.

4. Respect page stability
Core brand and informational sections should remain stable unless a user explicitly edits them or approves changes.

5. Keep auditability
Record:
- what changed
- why it changed
- what source triggered it
- whether it was auto-applied or pending review

6. Add confidence logic
If the system is not confident, it should create a suggestion instead of auto-publishing.

PART 7 — SOURCE OF TRUTH HIERARCHY

Establish this content precedence order:

1. Explicit admin-entered Pillar data
2. Structured platform records (events, sponsors, contacts, settings, announcements)
3. Approved imported content from source website
4. AI-generated fallback content
5. Generic placeholder content only as last resort

This hierarchy is important.
AI should not overwrite strong source material or real structured data.

PART 8 — SITE IMPORT UX

When importing from a URL, the user experience should show what was found.

I want an import review summary such as:
- pages detected
- logo found
- contact info found
- programs detected
- events detected
- image count imported
- questionable items flagged
- recommended site structure

Let the user approve or edit the imported foundation before final generation.

PART 9 — IMPLEMENTATION GOAL

Please improve the importer and auto-update system so that:
- importing an existing site makes the generated result better, not messier
- the system uses the best parts of the old site intelligently
- the generated site keeps organization identity
- auto-update is modular, conservative, and trustworthy
- dynamic sections update from real data with minimal risk
- AI is used as an editor and organizer, not an uncontrolled rewriter

PART 10 — DELIVERABLES

Before implementing major changes, give me:
1. critique of the current site import behavior
2. proposed structured import pipeline
3. image ranking/import strategy
4. auto-update rules and content classification model
5. what will auto-update directly vs require approval
6. phase 1 implementation plan

Then implement the highest-value phase 1 version.

Focus first on:
- better site classification and structured extraction
- better logo/image handling
- preserving strong source copy
