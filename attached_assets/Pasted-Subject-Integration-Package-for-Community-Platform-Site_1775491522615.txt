Subject: Integration Package for Community Platform Sites

I'm sending you a package (pillar-integration-package.tar.gz) that contains two folders:

Folder 1: pillar-integration/ — 4 files

README.md — Start here. Explains both integration approaches and lists every file in the package.
SETUP-API-SPEC.md — The full API spec for POST /api/pillar/setup. Documents the endpoint URL, the authentication header (X-Pillar-Key), every field in the JSON payload with types and defaults, the response format, and error codes.
example-payload.json — A complete, working example payload that provisions a site as "Norwin Rotary Club." Every field is filled in. I tested this against the live endpoint and it works.
interview-to-payload-map.md — A table that maps each interview question to the exact JSON field name in the setup payload. Includes how to generate the stats array (always 4 items), how to pick colors by org type (hex values for all 10 org types are listed), and which fields to derive from answers vs. ask directly.
Folder 2: community-platform-framework/ — 10 files

BUILDER-INSTRUCTIONS.md — The master build guide. Contains the Feature Decision Matrix (which pages, homepage sections, event detail sections, and features to include or skip based on org type and interview answers). Also covers the crawl-first workflow, security patterns (honeypot + timing on every form), admin dashboard structure, and the full tech stack.
INTAKE-QUESTIONS.ts — All 24 interview questions. Each one has the question text, an example answer, and which config field it maps to. Also contains ORG_TYPE_COLOR_PALETTES — the default primary/accent colors for each of the 10 org types.
org-config.ts — The TypeScript interface for OrgConfig plus a complete example filled in with IBPA (Irwin) data.
schema.ts — The full Drizzle ORM database schema. All tables: users, events, businesses, sponsors, vendor registrations, sponsorship inquiries, ticket purchases, blog posts, newsletter subscribers, site content, org config.
storage-interface.ts — The complete IStorage interface with every CRUD method the app uses.
api-routes-pattern.ts — All Express route handlers: public endpoints, auth endpoints, admin CRUD endpoints. Copy-ready.
frontend-patterns.tsx — Every major React component: HeroSection, StatsSection, EventCard, EventDetailPage, TicketPurchaseSection, SponsorshipSection, BlogListPage, BlogPostDetail, RecentPostsSection, NewsletterSection, ContactPage, PhotoGallery, Navigation, Footer.
theme-variables.css — CSS custom properties for light and dark mode. Replace --primary and --accent with the org's colors.
content-hooks.ts — The webhook notification system. When events are activated, tickets sell, blog posts publish, or sponsors are added, the site fires a webhook to Pillar with a strategy payload (priority, suggested platforms, tone, cadence limits) so your autopilot can generate and schedule social media posts.
registration-window-engine.ts — Auto open/close logic for vendor and sponsor registration. Opens 90 days before event, closes 7 days before, with admin force-open override.
What your AI needs to do:

Run the interview from INTAKE-QUESTIONS.ts. If the customer gives you their existing website URL, crawl it first and pre-fill answers — only ask what the crawl missed.
Use the Feature Decision Matrix in BUILDER-INSTRUCTIONS.md to decide which features this org needs. A Rotary Club doesn't get a business directory. A PTA doesn't get vendor registration. An org with no ticketed events doesn't get payment integration.
Map the interview answers to the setup API payload using interview-to-payload-map.md.
Call POST /api/pillar/setup on the deployed site with the payload.
The site rebrands instantly — colors, org name, logo, stats, partners, footer text, hero image, events, sponsors, businesses, and all CMS content.
What you need to set on each deployed site:

PILLAR_SERVICE_KEY — The auth key your AI sends in the X-Pillar-Key header when calling the setup API. Generate a unique one per site.
DATABASE_URL — PostgreSQL connection string.
SESSION_SECRET — Random string for Express session encryption.
SENDGRID_API_KEY — Only if the org has newsletter signup or ticketed events.
SQUARE_ACCESS_TOKEN, SQUARE_LOCATION_ID, SQUARE_WEBHOOK_SIGNATURE_KEY — Only if the org sells tickets via Square.
That's the complete message. You can copy-paste it to Pillar as-is.
