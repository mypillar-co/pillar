Tell Replit:
Create a new file artifacts/api-server/src/scripts/e2e-healthcheck.ts that runs a complete end-to-end health check of the entire Pillar platform. This script hits real endpoints against localhost, logs pass/fail for every step, and exits with code 1 if anything fails.
The script should check every critical flow in this order. Use fetch() for all HTTP calls. Log each step as PASS or FAIL with the actual vs expected result. Never throw on failure — catch every error, log it, and continue to the next test so you get a complete picture of what is broken.
Here is the complete list of flows to test:
FLOW 1 — API SERVER HEALTH
1a. GET /api/healthz → expect { status: “ok”, db: “ok” }
1b. GET /api/healthz on community platform port 5001 → expect { status: “ok” }
FLOW 2 — ORGANIZATION RESOLUTION
2a. GET /api/organizations (with valid auth header using SERVICE_API_KEY) → expect array with at least one org
2b. Pick the first org. GET /api/community-site/target for that org → expect isProvisioned field present
FLOW 3 — COMMUNITY PLATFORM ROUTING
3a. GET http://localhost:5001/ → expect 200 and content-type text/html
3b. GET http://localhost:5001/sites/norwin-rotary-uic5/ → expect 200 and content-type text/html
3c. Check that the HTML response from 3b contains /sites/placeholder/assets/ — if it does not the base swap is broken
3d. GET http://localhost:5001/api/healthz → expect 200
FLOW 4 — MEMBER PORTAL PROVISIONING
4a. Query the database directly: SELECT id, name FROM organizations WHERE id IN (SELECT DISTINCT org_id FROM members) LIMIT 1 — get an org that has at least one member
4b. GET /api/community-site/target for that org → expect isProvisioned true
4c. GET http://localhost:5001/members on that org’s subdomain (use x-org-id header) → expect 200 not 404
4d. Query the database: SELECT COUNT(*) FROM members WHERE org_id = <orgId> AND status = ‘pending’ — report pending invite count
FLOW 5 — MEMBER AUTH FLOW
5a. Create a test member via POST /api/members with firstName “E2E”, lastName “Test”, email “e2e-test-{timestamp}@pillar-test.local”, memberType “general” — expect 201
5b. Check the database that the member was created with a registration_token set
5c. Check the database that portal_provisioned = true for that org after the member was added — if false log “PORTAL PROVISIONING FAILED” as a critical failure
5d. POST to the community platform /api/members/register with the token and password “TestPass123!” → expect 200 with ok: true
5e. POST to /api/members/login with email and password → expect 200 with session cookie
5f. GET /api/members/me with the session cookie → expect the member’s firstName back
5g. Clean up: DELETE the test member from the database directly
FLOW 6 — EVENTS
6a. GET /api/events for the test org → expect 200 and array (may be empty)
6b. POST /api/events to create a test event with name “E2E Test Event”, startDate tomorrow, isActive true → expect 201
6c. GET /api/events → expect the new event appears in the list
6d. GET the community platform /api/events with x-org-id header → expect 200 and the event appears
6e. DELETE or soft-delete the test event → expect 200
FLOW 7 — SITE CONFIG
7a. GET /api/community-site/target for the test org → log the full configSummary
7b. Check that primaryColor, accentColor, orgName, and location are all non-null — log any that are missing
FLOW 8 — AI EDIT ROUTE
8a. POST /api/community-site/ai-edit with changeRequest “change the tagline to E2E test tagline” → expect 200 with ok: true and a payload back within 45 seconds
8b. If it times out log “AI EDIT TIMEOUT” as a warning not a failure — the AI service may be slow
FLOW 9 — CONTENT STUDIO
9a. POST /api/content/generate with taskId “newsletter_intro” and valid inputs → expect 200 with output string
9b. GET /api/content/history → expect 200 and array
FLOW 10 — AUTOPILOT AGENT
10a. POST /api/management/agent with message “run a self test” → expect 200 and a response that does not contain “error”
10b. POST /api/management/agent with message “how many members do we have” → expect 200 and a response that contains a number
FLOW 11 — SOCIAL AND BUFFER
11a. GET /api/social/accounts → expect 200 (array may be empty if Buffer not connected)
11b. If accounts exist GET /api/social/posts → expect 200
FLOW 12 — MEMBER STATS AND RENEWAL REMINDERS
12a. GET /api/members/stats → expect 200 with total, active, board, pending fields
12b. Query the database: SELECT COUNT(*) FROM members WHERE renewal_date <= (NOW() + INTERVAL ‘30 days’) AND status = ‘active’ — report how many renewals are coming up
FLOW 13 — DEPLOY GATE
13a. Check that deploy-cp.sh exists at the repo root → FAIL if missing
13b. Check that artifacts/community-platform/dist/public/index.html exists → FAIL if missing
13c. Run grep -c “/sites/placeholder/assets/” artifacts/community-platform/dist/public/index.html → expect count > 0, FAIL if zero and log “BUILD BASE SWAP MISSING — CP WILL SERVE BLANK PAGES”
At the end of the script print a summary:
	•	Total tests run
	•	Total passed
	•	Total failed
	•	Total warnings
	•	List every FAIL and WARNING with its flow number and description
	•	Exit code 0 if zero failures, exit code 1 if any failures
