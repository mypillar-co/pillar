I want to change the Pillar website generator so it behaves more like a strong human UX/web designer.

This is not an architecture expansion task.
This is not a schema redesign task.
This is a decision-making / planning / design-judgment task.

The generator must follow a deterministic design algorithm that combines:
1. real platform data
2. imported website signals
3. constrained AI assistance
4. design restraint

GOAL

Pillar should stop behaving like “AI fills a template.”
It should behave like:
- a competent UX designer
- a disciplined template system
- a curator of good content
- a restrained planner that knows what to emphasize and what to hide

==================================================
PART 1 — CORE DESIGN PHILOSOPHY
==================================================

The system must behave the way a strong human UX designer would behave.

A strong designer does NOT:
- use every available image
- show every possible section
- promote weak content
- let bad assets dominate the page
- make every page do everything

A strong designer DOES:
- identify the page’s main job
- score content quality
- choose what deserves prominence
- hide weak content
- use simple layouts when confidence is low
- preserve identity but improve structure
- tell one clear story per homepage

The website generator must follow this same logic.

==================================================
PART 2 — THE DESIGN ALGORITHM
==================================================

The generator must follow this exact decision flow.

----------------------------------
STEP 1 — GATHER INPUTS
----------------------------------

Gather all available inputs:

A. platform data
- organization profile/settings
- events
- sponsors
- contacts
- announcements
- social links
- ticket/registration configuration
- public event behavior

B. interview/intake data
- mission
- audience
- programs/services
- membership/join language
- location/contact
- CTA preferences
- tone

C. imported website data
- existing page structure
- navigation labels
- mission statements
- about/program text
- event listings
- join/donate links
- sponsor/partner references
- hero image candidates
- logos
- contact information
- CTA patterns

D. available assets
- logo(s)
- photos
- graphics
- event images
- imported site images

----------------------------------
STEP 2 — CLASSIFY ALL ASSETS
----------------------------------

Every asset must be classified into one of these types:

- logo
- hero_photo
- supporting_photo
- decorative_graphic
- sponsor_logo
- icon_mark
- unknown

Hard rules:
- logos, decorative graphics, icon marks, sponsor logos, and unknown assets must NEVER be used as hero backgrounds
- only hero_photo may be used as hero background
- if confidence is low, downgrade the asset instead of promoting it

Classification heuristics:
- transparent background, square, vector-like, flat brand art → logo/decorative/icon_mark
- very small image or poor resolution → not hero_photo
- landscape photographic image with enough width and no major text baked in → hero_photo candidate
- repeated logo treatment across imported site → logo or sponsor_logo
- event flyer/poster with too much text → supporting_photo, not hero_photo

If no strong hero_photo exists:
use a non-image hero.

----------------------------------
STEP 3 — IDENTIFY THE PRIMARY JOB OF THE SITE
----------------------------------

The homepage must have one primary job and one optional secondary job.

Possible primary jobs:
- event_conversion
- membership_conversion
- program_explanation
- contact_capture
- credibility_showcase

Determine primary job using real signals:

event_conversion if:
- there are multiple public events
- there are ticketed/registration events
- event CTA is the strongest user action

membership_conversion if:
- join/membership language is strong
- mission is strong
- benefits/community/fellowship language is present

program_explanation if:
- there are multiple specific programs/services
- programs are richer than events

contact_capture if:
- content is sparse
- contact/location/action is the strongest available value

credibility_showcase if:
- sponsors/partners/history/legacy are the strongest content

Output:
{
  primaryJob,
  secondaryJob
}

----------------------------------
STEP 4 — SCORE CONTENT QUALITY
----------------------------------

Each content category gets a quality score:

0 = absent
1 = weak
2 = usable
3 = strong

Score these categories:
- mission
- events
- programs
- membership
- contact
- sponsors
- images
- announcements
- stats

Scoring rules:

Mission:
0 = none
1 = generic/vague
2 = clear but not highly specific
3 = specific audience, clear purpose, action-oriented, identity-rich

Events:
0 = none
1 = one vague or outdated event
2 = at least one real public event
3 = multiple real public events, dates, locations, CTA-ready

Programs:
0 = none
1 = generic mentions only
2 = at least 2 named programs/services
3 = multiple named, specific programs/services

Membership:
0 = no join language
1 = vague mention
2 = clear join/member/involvement language
3 = clear benefits, belonging, or conversion path

Contact:
0 = weak/missing
1 = partial
2 = usable
3 = clear email/phone/address/contact path

Sponsors:
0 = none
1 = weak/minimal
2 = some visible real sponsors
3 = strong sponsor/partner presence

Images:
0 = none
1 = weak/logo-only
2 = usable supporting imagery
3 = strong real hero/supporting imagery

Announcements:
0 = none
1 = weak/stale
2 = usable
3 = active/current

Stats:
0 = none
1 = vague or weak
2 = real but limited
3 = real and meaningful

Hard rule:
existence alone does not equal quality.

----------------------------------
STEP 5 — INCORPORATE IMPORTED WEBSITE SIGNALS
----------------------------------

Imported websites must influence the algorithm directly.

Imported data must affect:
- primary job selection
- content quality scores
- section prominence
- layout strategy
- CTA priority
- hero choice
- image classification confidence

Imported signals to detect:
- hero image present
- event listings or calendar pages
- ticket/buy/register links
- join/membership language
- donate language
- sponsor/partner pages
- strong mission statement
- strong nav structure
- repeated CTA pattern
- contact prominence
- gallery/photo richness

Rules:
- if imported site is clearly event-heavy, boost event score
- if imported site is clearly membership/join heavy, boost membership score
- if imported site has strong preserved mission text, boost mission score
- if imported site has good hero imagery, boost image score
- if imported site has sponsor/partner structure, boost sponsor score
- if imported site shows repeated ticket or registration behavior, boost event_conversion and CTA priority

Do NOT treat import as passive content only.
Import must shape strategy.

----------------------------------
STEP 6 — DETERMINE WHAT DESERVES PROMINENCE
----------------------------------

Prominence rules:

Score 3:
may be a leading homepage section

Score 2:
may be a supporting section

Score 1:
must be visually minimized or merged into another section

Score 0:
must be omitted

This is critical.
Weak content should not be promoted just because it exists.

----------------------------------
STEP 7 — CHOOSE A LAYOUT STRATEGY
----------------------------------

Choose exactly one strategy:

- event-led
- membership-led
- program-led
- contact-led
- credibility-led
- minimal

Selection rules:

event-led if:
- primaryJob = event_conversion
- events score is high
- or imported site strongly indicates event-heavy structure

membership-led if:
- primaryJob = membership_conversion
- mission + membership scores are both usable/strong

program-led if:
- primaryJob = program_explanation
- programs score is high

contact-led if:
- primaryJob = contact_capture
- content overall is sparse but contact is clear

credibility-led if:
- sponsors/history/partners are strongest trustworthy content

minimal if:
- most scores are weak
- there is not enough high-confidence material for a richer site

Do not choose based on org type alone.

----------------------------------
STEP 8 — CHOOSE HERO TYPE BY CONFIDENCE
----------------------------------

Choose hero type conservatively.

Possible hero types:
- featured-event hero
- mission hero
- photo hero
- clean text hero
- logo-badge hero

Rules:

Use featured-event hero if:
- strategy is event-led
- and there is a strong featured event or clear fallback event

Use mission hero if:
- mission score is strong
- strategy is membership-led or credibility-led

Use photo hero if:
- there is a true hero_photo
- and the image score is high

Use clean text hero if:
- content is clear but assets are weak

Use logo-badge hero if:
- logo is strong
- but there is no hero-quality photo

Hard rule:
If confidence is low, use clean text hero or logo-badge hero.
Do not force image hero.

----------------------------------
STEP 9 — BUILD SECTION ORDER USING PERSUASION LOGIC
----------------------------------

Homepage order must follow this persuasion sequence:

1. Hero
- communicate primary job immediately

2. Strongest proof
One of:
- featured event
- mission summary
- program summary
- benefits/join reasons

3. Supporting proof
One of:
- events list
- programs grid
- sponsor strip
- announcements

4. Trust / legitimacy
One of:
- sponsors/partners
- history/credibility
- active announcements
- community proof

5. Conversion
One clear CTA section:
- join
- register
- contact
- donate

6. Footer/contact

Do not allow random or repetitive section order.

----------------------------------
STEP 10 — APPLY RESTRAINT RULES
----------------------------------

Hard restraint rules:

- max 4–6 homepage sections unless content quality is unusually strong
- no more than 2 CTAs per section
- no fake stats
- no weak sponsor strip
- no hero background from logo/decorative asset
- no gallery unless images are actually good
- no event section if there are no public events
- no generic filler cards when content is too thin
- no complexity when a simple page would look better

The system must prefer:
fewer sections, better quality

==================================================
PART 3 — EVENT FLOW INTELLIGENCE
==================================================

The site must understand event behavior and render the right public experience.

Each event must map to a public mode:
- informational
- RSVP
- registration
- ticketed
- fundraiser
- vendor application
- participant signup
- sponsorship
- hybrid

Each event must have a public CTA mode:
- learn_more
- register
- rsvp
- buy_tickets
- apply_vendor
- apply_participant
- sponsor
- donate

Site rules:
- informational events → Learn More
- registration events → Register / RSVP / Sign Up
- ticketed/fundraiser events → Buy Tickets / Reserve Seats
- vendor events → Apply as Vendor
- participant events → Sign Up / Participate
- sponsor opportunities → Become a Sponsor

Featured event fallback:
1. explicitly featured event
2. next ticketed/fundraiser event
3. next public upcoming event
4. else no featured event block

Dynamic event blocks must:
- use live event data
- respect public visibility
- show sold out state
- show registration closed state
- show pricing only when allowed
- show CTA matching event mode

Event-led sites should prioritize:
Hero
Featured Event
Event List
CTA
Sponsors
About/Contact

Membership-led sites should prioritize:
Mission Hero
Benefits/Join
Events secondary
Contact

==================================================
PART 4 — IMPORTED WEBSITE LOGIC
==================================================

Imported sites must not just provide text.
They must influence structure and judgment.

The importer must extract:
- nav structure
- page purposes
- mission statement
- join/donate language
- event/ticket/registration signals
- sponsor/partner references
- hero/photo candidates
- contact prominence
- design cues where possible

Imported content handling rules:
- preserve strong mission verbatim
- preserve strong program naming
- preserve strong CTA language if specific
- preserve logo
- improve layout
- do not flatten a rich site into a generic brochure
- do not reword strong identity content into generic AI fluff

Imported site outcome should feel like:
“we kept what mattered and rebuilt it better”

==================================================
PART 5 — VISUAL DESIGN RULES
==================================================

Apply these consistently:

1. Never use logos as hero backgrounds
2. Use max page width around 1100–1280px
3. Use strict typography scale
4. Use consistent spacing scale
5. Limit cards per row
6. Keep heroes to one focal point
7. If content is weak, simplify
8. If image confidence is low, do not use it as hero
9. Use clean CTA hierarchy
10. Default to legibility over cleverness

==================================================
PART 6 — HOW TO APPLY THIS TO PILLAR
==================================================

Implement this as deterministic logic in:

- siteProfileService
- import signal extraction
- layout signal scoring
- pagePlanner
- hero/image selection logic
- block rendering decisions
- event behavior mapping

AI may only:
- help extract structured signals
- help generate structured block copy
- help summarize imported content

AI may NOT:
- decide layout freely
- decide asset roles freely
- override deterministic design rules
- invent hero usage from weak assets
