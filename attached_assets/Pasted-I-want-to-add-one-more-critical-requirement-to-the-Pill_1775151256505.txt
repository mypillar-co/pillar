I want to add one more critical requirement to the Pillar website generator, event system, and importer.

The generated site must handle events, vendors, participants, registrations, and sales dynamically and correctly.

This is not optional brochure content.
For many organizations, events are the website.

The system must understand the operational type of each event and render the right site behavior automatically.

CORE GOAL

When Pillar generates or imports a website, it should correctly model and display:
- events
- ticketed events
- free registration events
- vendor registration events
- participant signup events
- sponsor-driven events
- donation/fundraising events
- informational events with no registration
- events that mix multiple flows

The website must be dynamic and tied to real platform data, not just static text.

PART 1 — EVENT TYPE INTELLIGENCE

Add an event-type interpretation layer.

For each event, the system should infer and/or explicitly store:

- event purpose
- whether it is public-facing
- whether it is informational only
- whether it requires RSVP
- whether it requires participant registration
- whether it requires vendor registration
- whether it has sponsorship opportunities
- whether it is ticketed
- whether it is a fundraising/profit event
- whether it is just a registration/attendance event
- whether it needs capacity tracking
- whether it needs payment status tracking
- whether it has deadlines for vendors/participants/sponsors

Examples:
1. Rotary dinner fundraiser
   - public event
   - ticketed
   - fundraising/profit event
   - sponsor visibility likely relevant
   - ticket sales and revenue should be surfaced

2. Community volunteer cleanup
   - public event
   - registration event
   - not a profit event
   - no ticket sales
   - participant signup needed

3. Street market
   - public event
   - vendor registration event
   - may have sponsor opportunities
   - may or may not have public ticketing
   - vendor approvals and vendor sales tracking matter

4. Parade participation signup
   - participant registration event
   - not ticketed
   - capacity / form collection matters
   - no “tickets sold” language

The AI and the platform should understand these differences and not treat all events the same.

PART 2 — TOGGLEABLE SALES / REGISTRATION BEHAVIOR

Ticket sales must be toggleable at the event level.

Each event should support clean configurable modes such as:

- informational only
- RSVP only
- free registration
- paid ticket sales
- vendor application
- participant registration
- sponsor opportunities only
- hybrid mode (for example public attendance + vendor application)

Do not assume every event needs ticketing.

For every event, the system should be able to toggle:

- tickets enabled on/off
- registration enabled on/off
- vendor applications enabled on/off
- participant signup enabled on/off
- sponsorship enabled on/off
- public sales totals shown or hidden
- capacity limits on/off
- public countdown / urgency UI on/off
- waitlist mode on/off if needed later

PART 3 — UNDERSTAND REGISTRATION EVENT VS PROFIT EVENT

This is very important.

I need the AI and data model to distinguish between:

A. registration / participation events
Examples:
- volunteer signup
- parade participation
- membership interest night
- board dinner RSVP
- free community event with registration
- vendor application intake where fees are administrative but the event is not fundamentally ticket-sales-driven

B. profit / fundraising events
Examples:
- gala
- dinner fundraiser
- paid community night
- ticketed benefit event
- festival ticket sales
- events where revenue, sales, capacity, and conversion matter

This distinction should affect:
- site UI
- dashboard metrics
- copy tone
- CTA language
- reporting
- whether revenue widgets appear
- whether “tickets sold” is emphasized
- whether “register” vs “buy tickets” is used

Examples:
If it is a registration event:
- CTA should say Register / Sign Up / Participate
- show registration status
- avoid revenue-heavy language
- focus on attendance, participation, form completion

If it is a profit event:
- CTA should say Buy Tickets / Reserve Seats / Purchase Admission
- show pricing, availability, sold count if enabled
- emphasize fundraising/revenue if appropriate

The AI should infer this from:
- imported source site language
- event metadata
- form/payment setup
- admin-configured settings
- organization type
But admins must also be able to override the classification.

Build reusable modules for:
- upcoming events list

- registration event card

- participant signup section
- sponsor opportunities section
- event countdown / deadline notice
- sold out / registration closed / force open states
- capacity remaining indicator if enabled
- event detail page modules

The site should bind these modules to real platform records.

If event data changes in Pillar, the site should update the affected module appropriately.

- vendors

Do not collapse all money into one generic “sales” number.

Track and differentiate:
- ticket revenue
- vendor fee revenue
- participant fee revenue
