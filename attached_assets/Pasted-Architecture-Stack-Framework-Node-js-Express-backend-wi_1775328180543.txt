Architecture & Stack
Framework: Node.js + Express backend with a React (Vite) frontend, using TypeScript throughout.

Why this stack:

Express provides a lightweight, flexible server for API routes, authentication, and email sending
React with Vite gives fast hot-reloading during development and a modern component-based frontend
TypeScript catches errors at build time and keeps the data model consistent between frontend and backend
This stack runs natively on Replit with zero external infrastructure
Database: PostgreSQL with Drizzle ORM

Why:

PostgreSQL is provided by Replit out of the box — no setup or external service needed
Drizzle ORM provides type-safe queries that match the schema defined in shared/schema.ts, so the frontend types and database schema are always in sync
Drizzle-zod generates validation schemas automatically from the database model
Routing: Wouter (frontend), Express (backend)

Why wouter over React Router:

Tiny footprint (~1.5KB), no context providers needed
Simple API that matches this project's needs without overhead
Design System

Primary (Midnight): #12233e — used for headers, navbar, primary buttons
Accent (Cornflower): #5b7db1 — links, highlights, icons
Background (Parchment): #f7e8e0 — warm off-white background
Secondary: #ded5cd — card backgrounds, subtle sections

Why:

Accessible by default (keyboard navigation, screen readers, ARIA attributes)
Unstyled primitives that accept custom theming — so the PA Masons color palette applies cleanly
No heavy component library dependency — components are copied into the project and owned by you
Authentication & Security
Session-based auth with Passport.js + express-session:

Sessions stored in PostgreSQL (connect-pg-simple) so they survive server restarts
Passwords hashed with scrypt (Node.js built-in, no bcrypt dependency)
Email-based 2FA:

On login, a 6-digit code is emailed via SMTP (Gmail) with a 10-minute expiry
Codes stored in a verification_codes database table, automatically cleaned up
Dev-only bypass for the test admin account to avoid needing email during development
Whitelist-based registration:

Only email addresses pre-added by an admin can register
Prevents unauthorized signups — important for a members-only lodge area
reCAPTCHA v2 on public forms:

Contact form and registration form protected against bots
Server-side verification of the reCAPTCHA token
Admin protection:

isAdmin boolean on the user model
AdminRoute component wraps all admin pages with auth + admin checks
All admin API routes wrapped in try/catch with role verification
Admins cannot remove their own admin status (self-protection)
Page Structure & Content Decisions
Public pages (accessible to anyone):

Home, About, Officers, Past Masters, Community, Events, Hall Rental, Calendar, Contact, FAQs, Becoming a Mason, Widows & Helpers
Members-only pages (behind authentication):

Dashboard with personalized greeting ("Bro. [Name], PM" if applicable)
Announcements, Events (with JotForm registration links), Members Calendar, Monthly Notices archive, Dues Info, Profile/Account
Admin panel (admin users only):

Manage Users (whitelist, toggle admin), Announcements, Events, Master's Messages, Monthly Notices, Documents, Officers
Why this split:

Public pages serve prospective members and the community (hall rental inquiries, event info)
Members area provides lodge-specific info that shouldn't be public (calendars, notices, announcements)
Admin panel lets the Secretary or Worshipful Master manage content without touching code
Event Registration Strategy
JotForm iframe embeds instead of built-in forms:

JotForm already integrates with Google Sheets for tracking registrations
No need to build, maintain, and debug custom form handling, email notifications, and export features
The iframe is sandboxed — it can't crash the React app (we learned this the hard way when script embeds caused DOM conflicts)
Time-gated registration:

Each event has REGISTRATION_OPEN and REGISTRATION_CLOSE dates
Before open: shows countdown
During window: shows JotForm
After close: shows closed message with contact info
All automatic, no manual toggling needed
Calendar Integration
Google Calendar API:

Public calendar for community events
Members-only calendar for degree work, practices, etc.
Events fetched server-side using the Google API key, cached and served to the frontend
Two separate Google Calendar IDs for public vs. members content
Email System
Nodemailer with Gmail SMTP:

Contact form submissions sent to shidlelodge@gmail.com
2FA codes sent from shidlelodge601@gmail.com
Hall rental inquiries routed to the secretary's email
Content Management Philosophy
Database-driven where content changes frequently:

Master's Messages (monthly archive, 13+ months seeded)
Announcements
Member events
Monthly notices
Officers list
Documents (Google Drive links)
Hardcoded where content is stable:

Page layouts, event details (Past Masters Night, Golf Outing)
Community partners
FAQs
About/history text
This way the admin can update the things that change (announcements, messages) without a developer, while structural content stays in version control.

Mobile Responsiveness
Tailwind CSS responsive utilities throughout (md:, sm: breakpoints)
Mobile hamburger menu with full navigation
Cards and grids collapse to single column on small screens
JotForm iframes set to 100% width with generous min-height (1200px) to show full forms without internal scrolling
Error Handling & Stability
safeLogout wrapper catches all promise rejections from the logout flow
Early inline script in index.html suppresses unhandled rejections that would crash the Replit workflow
All admin API routes wrapped in try/catch
Protected routes redirect to login with loading states
AdminRoute component handles loading, auth check, and admin verification before rendering
Deployment
Runs on Replit's hosting with automatic HTTPS
Custom domain support via A record (34.111.179.208)
PostgreSQL database persists across deployments
Session store in PostgreSQL means users stay logged in after deploys
This is everything that went into the architecture and decision-making. Every choice was driven by keeping it maintainable for a non-technical lodge secretary, reliable without dedicated DevOps, and appropriate for a fraternal organization's online presence.
