I want to replace the current Pillar website generator with a better architecture.

This is not a request to “make prettier templates.”
This is a request to redesign the website generation system so it can produce modern, dynamic, functional websites for civic/community organizations while staying editable and trustworthy.

IMPORTANT:
Do not rebuild the whole platform.
Do not generate arbitrary AI HTML/CSS from scratch.
Do not keep the current “one giant fixed template with token replacement” approach.
Do not overbuild a full Webflow clone.

Instead, build a structured, modular website system with AI-assisted planning and content generation.

CORE PRODUCT VISION

Pillar should not be an “AI homepage generator.”
It should be a dynamic organization website system with AI-assisted setup, import, planning, and maintenance.

The value is:
- generate a credible modern site quickly
- preserve the organization’s identity
- connect the site to real operational data
- keep the site updated safely
- avoid generic AI junk

The website builder should work especially well for:
- Rotary clubs
- VFW posts
- chambers
- lodges
- PTAs
- nonprofits
- downtown associations
- local event organizations

HIGH-LEVEL ARCHITECTURE

I want the system split into 4 layers:

1. Organization Intelligence Layer
2. Site Planning Layer
3. Rendering / Design System Layer
4. Live Operations / Dynamic Content Layer

Do not combine these into one sloppy AI generation step.

PART 1 — ORGANIZATION INTELLIGENCE LAYER

Goal:
Understand what the organization is, what the site is for, and what operational data matters.

Inputs may include:
- AI interview answers
- imported current website
- uploaded logo/images
- org type
- events
- sponsors
- vendors
- participants
- contact settings
- announcements
- admin preferences

Build a structured organization/site profile.

I want a normalized structured output such as:
- org type
- mission
- audience
- tone
- core CTAs
- recurring programs/services
- event strategy
- sponsorship/vendor relevance
- dynamic data sources available
- static identity content
- brand assets available
- missing critical fields

Do not output HTML at this stage.
Do not output final marketing copy at this stage.
Output structured data only.

PART 2 — SITE IMPORT / EXTRACTION PIPELINE

The importer currently creates messy, generic results.
I want a structured import pipeline.

When a source website URL is provided, do this:

Stage A: Crawl and classify pages
Detect likely page types:
- home
- about
- programs/services
- events
- join/get involved
- contact
- sponsor/partner
- donate
- blog/news
- FAQ

Stage B: Extract structured signals
Extract and normalize:
- organization name
- tagline
- mission statement
- contact info
- address / meeting schedule
- page/navigation structure
- program descriptions
- event references
- sponsor/partner references
- CTA patterns
- logo
- important images
- color/brand cues

Stage C: Quality filter
Reject:
- duplicated navigation text
- cookie/privacy clutter
- stale boilerplate
- low-value footer junk
- broken formatting
- tiny or irrelevant images
- repetitive fluff

Stage D: Create import summary
Show a reviewable import summary:
- pages detected
- logo found
- contact info found
- events detected
- programs detected
- images imported
- issues/questions flagged
- recommended site structure

Stage E: Map into structured Pillar records
Do NOT rewrite the entire source site into generic AI content.
Use the source site as:
- content source
- structure clue source
- brand cue source
- image source

The rule is:
curate and map, not scrape and rewrite.

Do not assume every organization gets the same pages.

Create site archetypes based on org type.

Examples:

- Contact

- Home

- Home
- Mission
- Programs
- Events
- Get Involved
- Contact

D. HOA / Community Org
- Home
- About
- Events
- Announcements

The site planner should decide:
- recommended pages
- navigation structure
- primary CTA strategy
- which pages should be dynamic
- which page modules belong on each page
- homepage emphasis (mission-first, event-first, membership-first, etc.)

PART 4 — EVENT / OPERATIONS MODEL

This is critical.

The website must understand event operations, not just display event text.

Add a proper event behavior model.

Each event should support:
- event mode
- revenue classification
- public display behavior
- active workflows

Event mode examples:
- informational
- registration
- ticketed
- fundraiser
- vendor-driven
- participant signup
- sponsor-driven
- hybrid

Revenue classification examples:
- no revenue
- registration/admin fee
- ticket revenue
- sponsorship revenue
- donation/fundraising
- mixed

Each event should be able to toggle:
- show on public site
- enable ticket sales
- enable free registration
- enable vendor applications
- enable participant signup
- enable sponsorship section
- show pricing publicly
- show sold count publicly
- track capacity
- show countdown/deadline
- show approved sponsors
- show approved vendors if enabled

The system must distinguish between:
- registration/attendance events
- fundraising/profit events

This distinction must affect:
- CTA language
- event page layout
- metrics shown
- copy tone
- public modules rendered

Examples:
Registration event:
- Register / Sign Up / Participate
- focus on attendance and participation
- do not over-emphasize revenue

Fundraiser/ticketed event:
- Buy Tickets / Reserve Seats
- show pricing / availability / sold out states
- may emphasize fundraising results

Admin must be able to override AI suggestions.

PART 5 — LIVE OPERATIONS / DYNAMIC CONTENT LAYER

Generated websites must support dynamic modules tied to real platform data.

This is a core requirement.

Support dynamic modules for:
- upcoming events
- featured event
- event lists/calendar
- ticketed event cards
- registration event cards
- vendor application modules
- participant signup modules
- sponsor opportunity modules
- sponsor logo grids
- announcements
- volunteer opportunities
- contact / interest capture
- countdown / deadline notices
- sold out / registration closed states

If event/vendor/sponsor/announcement data changes in Pillar, update the module, not the whole site.

The site should behave like a live organization website, not a frozen generated brochure.

PART 6 — RENDERING / DESIGN SYSTEM LAYER

Do NOT generate arbitrary HTML/CSS from scratch.

Build a reusable, high-quality section/module system.

Create a library of strong site modules:
- hero variants
- mission/introduction blocks
- event-focused hero
- stats (only when real data exists)
- card grids
- timeline sections
- testimonial/community quotes
- sponsor strips
- gallery/media sections
- CTA bands
- FAQ sections
- contact sections
- join/volunteer/donate sections
- board/member highlight sections
- announcements feed blocks

Each module should support variants.
Example:
- hero: centered / split / image-right / image-background / event-emphasis
- card grid: 2-up / 3-up / icon-focused / image-focused
- CTA band: membership / donation / sponsor / event registration

AI should choose:
- which modules to use
- in which order
- which variant best fits the org/site plan

But within a curated design system.

PART 7 — CONTENT GENERATION RULES

AI-generated content must be constrained by quality rules.

Rules:
- preserve strong source copy if imported from an existing site
- preserve organization-specific language
- preserve real mission statements where strong
- preserve real program names
- preserve real contact/location/meeting info
- avoid generic nonprofit buzzwords
- never invent fake stats
- if real facts are missing, omit the section or mark it for completion
- prefer specificity over marketing fluff

The rule is:
enhance and organize strong material, do not replace identity with generic filler.

PART 8 — IMAGE / MEDIA STRATEGY

Stop relying on a tiny fixed pool of stock images.

Image priority order should be:
1. uploaded logo and organization-owned images
2. relevant imported source-site images
3. curated fallback imagery by org/event type
4. graceful placeholders

Add image ranking logic:
- logo detection
- hero-worthiness
- resolution/quality
- relevance to organization type
- duplicate detection
- decorative vs content-bearing images

Do not clutter generated sites with weak imported images.
Do not mix random junk images with premium design.

The system should still look good when no good photos exist.

PART 9 — AUTO-UPDATE RULES

If a user enables auto-update, the system must be conservative and modular.

Do NOT regenerate entire pages automatically.

Auto-update must operate at the section/module level.

Safe to auto-update:
- upcoming events
- event status / deadlines / sold out states
- sponsor logo grids from approved sponsors
- public vendor/participant states if configured
- announcements from structured records
- contact info from settings
- simple dynamic counters based on real data

Suggest for review, do not auto-publish:
- featured event promo text
- homepage announcement banners
- seasonal CTA copy
- AI-generated event summaries
- homepage “what’s new” blurbs

Never auto-rewrite without approval:
- mission
- about
- history
- programs/services descriptions
- identity copy
- legal/policy copy
- board/member bios
- major layout changes

Add a change log for auto-updates:
- what changed
- why it changed
- source of truth
- confidence
- auto-applied vs pending review

PART 10 — SOURCE OF TRUTH HIERARCHY

Establish this precedence order:

1. explicit admin-entered settings/data
2. structured platform records (events, sponsors, announcements, contact, etc.)
3. approved imported source-site content
4. AI-generated suggested content
5. placeholder content as last resort only

AI must never overwrite explicit admin configuration.

PART 11 — DATA MODEL / STORAGE DIRECTION

Refactor away from the “single generated HTML blob” model.

Move toward structured website records, for example:
- site_pages
- site_sections
- site_navigation
- site_theme or brand_settings
- content_blocks
- section_bindings (for dynamic data like events/sponsors/announcements)
- import_runs / import_findings
- site_change_log

The site should be editable after generation.

Admins should be able to:
- edit page copy
- reorder sections
- toggle sections on/off
- change hero variant
- manage navigation
- update CTA copy
- manage colors/theme presets
- review suggested changes
- bind/unbind dynamic modules

Do not build a giant no-code builder yet.
But build the data model so the site is no longer trapped in one HTML template.
