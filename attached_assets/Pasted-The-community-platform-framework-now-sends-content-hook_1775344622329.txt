The community platform framework now sends content hook webhooks with strategy metadata. Wire these into the existing social posting automation.

What's coming in:
Every framework-built site will POST to a webhook URL (stored in config.pillarWebhookUrl) whenever something happens — new event, blog published, ticket milestone, sold out, sponsor added, etc. Each POST includes a strategy object with priority, category, suggested tone, platform hints, cadence limits, and timing guidance.

What you need to build:

Webhook receiver endpoint — e.g. POST /api/hooks/site-event. Receives the payload, identifies which org it's from via orgName/orgWebsite, and queues it for processing.
Map hook categories to existing strategy rules — The payload has a strategy.category field (announcement, promotion, update, milestone, recognition, internal). Map these to the org's existing posting strategy rules. If the strategy says "post 3x/week" and 4 hooks come in on Monday, queue the extras for later in the week.
Respect cadence controls — Hooks include cadenceKey and cadenceLimitPerDay. Track these per org. Example: ticket milestones have cadenceKey: "ticket_milestone" with limit 1/day. If two milestones fire the same day, skip the second.
Use suggestedTone for copy generation — When the AI writes the social post, pass strategy.suggestedTone as a prompt instruction. It adapts per context: blog recaps get "reflective, celebratory", press releases get "professional, newsworthy", near-sellout milestones get "urgency, scarcity."
Priority routing — urgent = post immediately (sold out events). high = next in queue (new events, blog posts). normal = schedule per strategy (milestones, sponsors). low/internal = don't post, just log for analytics.
postImmediately: true overrides scheduling — Date changes and sold-out events set this flag. Skip the queue and post right away.
Platform filtering — suggestedPlatforms says which connected accounts to post to. Only post to platforms the org actually has connected. If the hook says ["x", "facebook"] but only X is connected, just post to X.
Image handling — When includeImage: true, pull the image from data.imageUrl or data.coverImageUrl. If no image URL is provided, use the org's default branding or skip the image.
Thread support — When threadWorthy: true and the platform is X, consider generating a thread instead of a single tweet for substantial content.
pillarWebhookUrl provisioning — When Pillar builds a new site, set config.pillarWebhookUrl to this endpoint URL. When no Pillar plan, leave it unset (hooks silently skip).
Hook events to expect:

event.activated — announce new event
event.updated — date/time/location changes only
ticket_sales.opened — tickets on sale
ticket_sales.milestone — 10/25/50/75/100/etc sold
ticket_sales.sold_out — capacity reached
blog.published — new post live
sponsor.added — thank/welcome sponsor
ticket.purchased — internal only, don't post
vendor.registered — internal only, don't post
The framework handles all the intelligence on what to send and when. Pillar's job is to receive it, run it through the existing strategy/automation rules, generate the copy, and post to connected accounts. The strategy object is a suggestion layer — Pillar's automation rules are always the final authority.
