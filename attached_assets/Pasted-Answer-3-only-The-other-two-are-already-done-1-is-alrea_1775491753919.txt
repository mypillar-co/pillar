#3 is what I need you to do. Here is exactly what Steward needs to do, step by step:

Step 1: Detect that the customer wants a community organization website.
When a customer's answers indicate they are a community org (Main Street association, Rotary, Chamber, VFW, PTA, Lions Club, arts council, neighborhood association, etc.), Steward enters this flow.

Step 2: Check if they have an existing website.
Ask: "Do you have an existing website?" If yes, crawl it with Cheerio. Extract: org name, tagline, address, phone, email, social links, about text, event listings, business listings. Store these as pre-filled answers.

Step 3: Run the interview.
The 24 questions are in INTAKE-QUESTIONS.ts in the package. Skip any question the crawl already answered. The questions cover:

Org identity: full name, short name, tagline, org type (dropdown of 10 types)
Location: city, state, physical address, mailing address
Contact: phone, email, events email, Facebook URL, Instagram URL
Logo: 2-3 letter abbreviation for the logo badge
Stats: annual events count, annual attendees count, local businesses count
Features: sponsors yes/no, vendors yes/no, ticketed events yes/no, payment provider (Square or Stripe), blog yes/no, newsletter yes/no
Content: partners list, event categories
Step 4: Build the JSON payload.
Map each answer to the exact field in the setup API payload. The field-by-field mapping is in interview-to-payload-map.md. Key transformations:

orgType answer → look up primaryColor and accentColor hex values from the color table in interview-to-payload-map.md. Example: "Rotary Club" → primaryColor: "#003DA5", accentColor: "#d4a017".
Stats answers → build a 4-element array: [{value: "15+", label: "Annual Events"}, {value: "1,000+", label: "Annual Attendees"}, {value: "50+", label: "Local Businesses"}, {value: "100%", label: "Volunteer Run"}]. For non-business orgs, replace "Local Businesses" with "Active Members".
Partners answer → parse into array of {name, description, website} objects.
Generate mission (expand tagline to 1-2 sentences), footerText (short org description), and metaDescription (SEO summary).
If the customer mentioned specific upcoming events, include them in the events array with title, description, date, time, location, category, and ticketing fields.
Build the siteContent object with keys: home_tagline, home_intro, home_subtitle, contact_address, contact_phone, contact_email, social_facebook, social_instagram, about_mission, community_partners.
Step 5: Call the API.

POST https://<deployed-site-domain>/api/pillar/setup
Header: X-Pillar-Key: <the PILLAR_SERVICE_KEY env var set on that site>
Header: Content-Type: application/json
Body: <the JSON payload from Step 4>

A complete working example payload is in example-payload.json in the package. It provisions a site as "Norwin Rotary Club" with Rotary blue/gold colors, 2 events, 2 sponsors, 4 partners, 4 stats, and CMS content.

Step 6: Confirm success.
The API returns {"ok": true, "orgConfig": {"id": 1, "orgName": "...", "shortName": "..."}, "seeded": {"events": 2, "sponsors": 2, ...}}. The site is now rebranded. No deployment, no build step, no code generation needed. The frontend reads /api/org-config on page load and applies colors, name, logo, stats, partners, and all content dynamically.
