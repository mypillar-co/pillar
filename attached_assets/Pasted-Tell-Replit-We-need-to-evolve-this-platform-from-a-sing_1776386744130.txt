Tell Replit:
We need to evolve this platform from a single-Replit prototype into a production-grade multi-tenant SaaS. Here is the full architectural roadmap, prioritized by impact:
1. Split the monorepo into independently deployable services
The three processes (API server, Community Platform, Steward) should each deploy independently. A crash or bad deploy to the CP should never take down the API or Steward. Move to separate Replit deployments or migrate to Railway/Render with separate services per process. Add a health check endpoint to each and a simple uptime monitor (Better Uptime or UptimeRobot).
2. Add a proper CI/CD pipeline
Right now every change goes straight to production. Add a staging branch that deploys to staging.mypillar.co. PRs merge to staging first, get smoke-tested, then promote to main. Never deploy the community platform without first verifying the built index.html contains /sites/placeholder/ in asset URLs. Add a pre-deploy check script that fails the build if that string is missing.
3. Protect the community platform build
Create a deploy-cp.sh script that: runs pnpm exec vite build, verifies the output index.html contains /sites/placeholder/assets/, then and only then restarts the CP process. Never allow vite.config.ts or server/index.ts to be edited without running this script. Add a PROTECTED_FILES list to the repo README.
4. Move sessions to Redis
connect-pg-simple works but PostgreSQL is not a session store — it adds latency and table bloat. Add a Redis instance (Upstash has a free tier) and switch express-session to connect-redis. Session reads drop from ~5ms to ~0.3ms. This also enables horizontal scaling later.
5. Add a job queue for async work
The scheduler running inside the API server process is a single point of failure and blocks the event loop during heavy runs. Move all async work (social post publishing, automated site updates, email sends, document reminder emails) to a proper job queue. Use BullMQ with the same Upstash Redis instance. Each job type gets its own queue with retry logic, dead letter queue, and per-job logging.
6. Fix the slug problem permanently
Every org gets a URL like ligonier-beach-w47h.mypillar.co. This looks unprofessional. On org creation, generate a clean slug from the org name with no random suffix. Add a uniqueness check — if ligonier-beach is taken, try ligonier-beach-2, not ligonier-beach-w47h. Store the suffix-free slug as the default. The random suffix should only be a fallback if the clean slug is genuinely taken after 5 attempts.
7. Add per-org Buffer OAuth
Right now all orgs share your personal Buffer API key. Each org needs to connect their own Buffer account via OAuth. Add BUFFER_CLIENT_ID and BUFFER_CLIENT_SECRET to secrets. Add GET /api/social/oauth/buffer/start → redirects to Buffer OAuth. Add GET /api/social/oauth/buffer/callback → exchanges code, encrypts token, stores in social_accounts with platform: "buffer". Update GET /api/social/buffer/profiles to use the org’s stored Buffer token instead of the global key. Update publishViaBuffer() in scheduler.ts to decrypt and use the org’s token.
8. Add real observability
Add Sentry to all three processes for error tracking. Add structured logging with correlation IDs so a single request can be traced across API server → CP. Add a /api/admin/health endpoint that checks DB connectivity, CP reachability, Redis ping, and returns a JSON status object. Set up alerts for error rate spikes and P95 latency over 2 seconds.
9. Add member management
Civic orgs live by their roster. Add a members table: id, orgId, firstName, lastName, email, phone, memberType, joinDate, renewalDate, status, notes. Add CRUD endpoints under /api/members. Add a Members page in Steward with import from CSV, search, filter by status, and bulk email. Add member count to the org stats. This is the single highest-value missing feature for the target customer.
10. Add a proper email system
The Communications tab in EventDetail calls api.events.communications.send() but the backend may only be logging. Verify Resend is actually sending. Add email templates per org using their brand colors from cs_org_configs. Add an email history table. Add unsubscribe handling. Add a bulk email tool that pulls from the unified contact list (vendors + sponsors + ticket buyers + members) with per-segment filtering.
11. Add tenant analytics
Each org needs to know who’s visiting their site and registering for events. Add a lightweight analytics table: page_views (orgId, path, referrer, timestamp). Inject a 1x1 pixel tracking endpoint into every community platform page. Add a simple analytics dashboard in Steward showing weekly visitors, top pages, registration conversion rate, and ticket sales trend. Do not use Google Analytics — keep it first-party and privacy-respecting.
12. Harden the database
Add connection pooling via PgBouncer (Neon supports this natively — enable it). Add database-level row security so a bug in the API can never leak one org’s data to another. Add orgId foreign key constraints everywhere they’re missing. Add a nightly backup verification job that restores to a shadow DB and runs a row count check.
13. Add a Pillar admin panel
There is no way to manage the platform itself without going directly to the database. Add a /admin route in Steward gated by users.is_admin = true. Show: all orgs with their tier, slug, site status, last login, MRR. Add the ability to impersonate an org (set x-admin-impersonate-org header), manually provision a site, change a tier, and view error logs per org. This is essential for support and sales.
14. Add automated testing
The most dangerous thing about this codebase right now is that there are zero tests. One bad edit to app.ts takes down every tenant. Add Vitest for unit tests on the critical path: slug generation, base URL swap, token encryption/decryption, payment categorization. Add Playwright for two end-to-end tests: (1) org signup → interview → site live, (2) event create → public page loads. Run both in CI before any deploy.
Do these in order. Items 1-4 are infrastructure safety. Items 5-8 are reliability. Items 9-11 are product value. Items 12-14 are scale readiness. The platform is genuinely good right now — these changes make it defensible at scale.​​​​​​​​​​​​​​​​
