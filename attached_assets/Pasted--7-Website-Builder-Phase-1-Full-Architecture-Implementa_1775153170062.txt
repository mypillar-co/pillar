#7 - Website Builder Phase 1 — Full Architecture Implementation

What & Why

Turn Pillar from a single-template HTML blob generator into a structured, dynamic website operating system. The system is built as seven distinct layers. The critical new addition over previous plans is the Site Data Integration Layer — a mandatory adapter contract between dashboard tables and the renderer. The renderer must never directly query dashboard tables (events, sponsors, vendors, contacts, etc.). All data flows through: dashboard tables → site adapters → site_data_sources → site_block_bindings → renderer.

generated_html is kept as a compiled artifact for preview/publish compatibility only. It is never edited directly and never a source of truth.

Non-negotiable principles

Structured records are the source of truth
generated_html is compiled output only — never edited directly
Import must curate and map — not scrape and rewrite
Site planning must happen before rendering
AI assists extraction, planning, and drafting of structured content — it does not write HTML or CSS
Event behavior must be explicit — not inferred ad hoc throughout the app
Auto-update operates at the block/module level only
Identity content (mission, about, history, programs) remains stable unless explicitly changed
Dashboard data must never connect directly to the renderer
All website data integrations must flow through the site data integration layer
Done looks like

Generated sites are composed from structured records in site_pages, site_blocks, site_nav_items, site_themes
generated_html is compiled from those records by siteCompiler.ts
Different org types produce different page plans (pages, section order, variants)
Events render with correct CTA language, ticket links, and sold-out states based on explicit event fields
Sponsor grid auto-updates from approved sponsors via the sponsor adapter
The renderer has no direct queries to dashboard tables — all data comes through adapters and site_data_sources
Stats are never fabricated — sections omit when data is absent
Auto-update writes to site_change_log for every change and touches only auto_apply blocks
Identity copy remains stable after generation
Site import stores structured findings in site_import_runs and site_import_findings, with user review before application
Out of scope for Phase 1

Drag-and-drop visual block editor UI
Full revision history UI beyond site_change_log
Public-facing vendor/participant roster modules (data model and adapters are set up; rendering is Phase 2)
Exhaustive multi-page crawler (Phase 1: homepage + up to 5 detected sub-pages)
Complex visual design controls beyond theme presets
Custom domain routing changes
Approvals/board links on the public site (stays internal)

6. Write all adapter services

Each adapter normalizes one dashboard data source into a clean, renderer-facing payload. The renderer calls adapters through siteDataIntegrationService.ts — it never imports from adapters directly.

eventSiteAdapter.ts Function: getEventSiteData(orgId, queryConfig, metrics): SiteEventItem[] Queries eventsTable using queryConfig (filters: showOnPublicSite, eventStatus, featuredOnSite, etc.; limit; orderBy). For each event, calls getEventBehavior() to get explicit CTA mode. Reads event_public_metrics for capacity/sold data. Returns: { id, slug, name, date, time, location, description: siteSummaryOverride ?? description, imageUrl: siteImageOverride ?? imageUrl, ctaLabel, ctaUrl, eventMode, showPricing, price, isSoldOut, isRegistrationClosed, siteDisplayVariant, featuredOnSite }. Never exposes raw financial data — only the fields the renderer needs.

sponsorSiteAdapter.ts Function: getSponsorSiteData(orgId, queryConfig): SiteSponsorItem[] Queries sponsorsTable where siteVisible = true and status = 'active'. Orders by tierRank asc then siteDisplayPriority asc. Returns: { id, name, logoUrl, website, tierRank }.

vendorSiteAdapter.ts Function: getVendorSiteData(orgId, queryConfig): SiteVendorItem[] Queries vendorsTable where siteVisible = true. Returns: { id, name, siteCategory, siteDisplayPriority }. Phase 1: returns data but no public-facing block renderer is built yet — bindings are set up so Phase 2 can add the renderer without schema changes.

contactSiteAdapter.ts Function: getContactSiteData(orgId, queryConfig): SiteContactItem[] Queries contactsTable where siteVisible = true. Orders by siteDisplayPriority. Returns: { id, firstName, lastName, siteRole, siteBio, sitePhotoUrl }. Phase 1: used for the contact block's details (email, phone, address come from organizationSiteAdapter first).

announcementSiteAdapter.ts Function: getAnnouncementSiteData(orgId, queryConfig): SiteAnnouncementItem[] Queries studioOutputsTable where siteEligible = true and siteStatus = 'approved'. Orders by sitePublishPriority desc, then createdAt desc. Applies limit from queryConfig. Returns: { id, title, body, createdAt, siteBlockTarget }.

socialSiteAdapter.ts Function: getSocialHandles(orgId): SiteSocialHandle[] Queries socialAccountsTable where siteVisible = true and isConnected = true. Orders by siteDisplayOrder. Returns: { platform, accountName, siteLabelOverride }.

paymentSiteAdapter.ts Function: getEventPaymentSiteData(eventId): EventPublicMetrics Reads only from event_public_metrics — never joins ticketSalesTable directly. Returns precomputed metrics safe for public display.

organizationSiteAdapter.ts Function: getOrgSiteProfile(orgId): OrgSiteProfile Reads from organizationsTable. Returns site-specific fields with priority: siteContactEmail ?? contactEmail, siteMissionOverride ?? description, siteTagline, sitePublicAddress, sitePublicHours, sitePublicDescription, logo. This is the highest-priority content source — it outranks interview data and import findings.

12. Write lib/site/services/blockContentService.ts

Function: generateBlockContent(plan: PagePlan, profile: SiteProfile): Promise<BlockContentMap>

BlockContentMap: map of blockId (by blockType + variantKey key) to contentJson object.

This is the only service that calls the AI during generation. One AI call per generation run. The prompt receives the page plan structure, site profile, and preserved source copy. The prompt instructs:

"Output a JSON object where each key is blockType:variantKey and the value is the contentJson for that block. Do not output HTML."
"If founding year, member count, or event count was not explicitly stated in the interview, output null for those stat fields. Do not estimate."
"If a field is in preservedSourceCopy, use it verbatim — do not rewrite it."
"For program card descriptions, use the exact program names from the interview. Do not use phrases like 'brings community members together'."
"If you lack sufficient information for a block, output null for that block's key. Do not fabricate content."
Blocks that are entirely bound to live data (events_list, sponsor_grid, announcements, featured_event) receive an empty contentJson: {} — their content comes entirely from the binding at render time. Do not ask the AI to write content for those blocks.

Block content schemas:

hero: { headline, subheadline, ctaLabel, ctaUrl, ctaSecondaryLabel?, ctaSecondaryUrl?, eyebrow?, imageUrl? }
about: { heading, body, imageUrl?, sectionLabel }
stats: { stats: [{ value: string, label: string }] } — only included if profile.hasRealStats
cards: { heading, sectionLabel, items: [{ icon, title, description }] }
cta_band: { heading, body, ctaLabel, ctaUrl }
contact: { heading, intro, email, phone, address, hours, socialLinks: SiteSocialHandle[], cardHeading, cardBody, cardCtaLabel, cardCtaUrl }
faq: { heading, items: [{ question, answer }] }
membership/join/volunteer: { heading, body, steps?: string[], ctaLabel, ctaUrl }
13. Write lib/site/services/blockRenderer.ts

Function: renderBlock(block: SiteBlock, theme: SiteTheme, siteData?: SiteDataMap): string

Deterministic — same inputs always produce same output. Never calls AI. HTML is returned as a string. CSS is inlined per block using CSS custom properties from the theme record. All blocks are mobile-first responsive. Scroll-reveal classes are applied consistently. No Tailwind — plain <style> blocks, namespaced by block type.

Hero variants:

hero-centered: full-bleed background image or gradient, centered headline/subheadline/CTAs, eyebrow label
hero-split: left text column (60%), right image (40%), strong typography hierarchy
hero-mission: large display text only, no image, accent color treatment, org type label eyebrow
hero-event-featured: featured event date prominent, event name as headline, CTA from event's publicCtaMode
About variants: about-split (text left, image right), about-centered (centered text, image below), about-simple (text only)

Stats variants: stats-strip (dark bar, animated counters), stats-cards (card grid). Render only when content has non-null stat values — return empty string otherwise.

Cards variants: cards-2up, cards-3up, cards-icon (icon-forward), cards-image (image-forward)

Events list variants — reads from siteData[block.id] (pre-fetched event adapter output):

events-list-standard: date block + info row per event. For each event: uses ctaLabel from getCtaLabel(event.publicCtaMode), shows price if showPublicPricing, shows "Sold Out" badge if isSoldOut, shows "Registration Closed" if isRegistrationClosed, links to /events/:slug/tickets for ticketed/fundraiser events.
events-list-featured-first: first event large card, remainder compact rows
events-list-compact: minimal list with date and title only
Featured event variants — reads from siteData[block.id]:

featured-event-banner: full-width promotional strip with event date, name, CTA button
featured-event-card: card layout with image, date, description excerpt, CTA
Sponsor grid variants — reads from siteData[block.id]:

sponsor-grid-standard: uniform logo grid, renders logoUrl as <img> or sponsor name text as fallback
sponsor-grid-tiered: groups by tierRank, renders tier label header per group
Announcements variants — reads from siteData[block.id]
