 Here's exactly how a URL like /events/meet-your-neighbor-dinner becomes a rendered page in the browser:

It's NOT server-side rendered HTML. It's a client-side SPA.
There is no HTML file generated per event. The server sends one single HTML shell for every route, and React renders the page content in the browser using JavaScript.

Step 1: Every request hits Express
When a browser requests https://discoverirwin.com/events/meet-your-neighbor-dinner, Express receives it. The request doesn't match any /api/* route, so it falls through to the Vite/static handler.

In production, server/vite.ts serves the pre-built React app:

// Production: serve the built index.html for ALL non-API routes
app.use(express.static(path.resolve("dist/public")));
app.get("*", (_req, res) => {
  res.sendFile(path.resolve("dist/public/index.html"));
});
In development, Vite's dev server middleware intercepts and serves index.html with HMR (hot module replacement).

Either way, the browser receives the same index.html regardless of whether you visit /, /events, /events/meet-your-neighbor-dinner, or /contact. It's always the same HTML file.

Step 2: The HTML shell
The index.html is minimal — just a div and a script tag:

<!DOCTYPE html>
<html>
<head>
  <title>Discover Irwin</title>
  <script type="module" src="/src/main.tsx"></script>
</head>
<body>
  <div id="root"></div>
</body>
</html>
That's it. No event content, no dynamic HTML. The browser downloads the JavaScript bundle, which contains the entire React application.

Step 3: React router reads the URL
main.tsx mounts the React app into #root. Then wouter (the client-side router) reads window.location.pathname:

// App.tsx
<Switch>
  <Route path="/events/:slug" component={EventDetail} />
  // ...other routes
</Switch>
wouter sees the path is /events/meet-your-neighbor-dinner, extracts slug = "meet-your-neighbor-dinner", and renders the EventDetail component.

Step 4: EventDetail fetches data from the API
The component uses the slug to make API calls back to the Express server:

// client/src/pages/event-detail.tsx
export default function EventDetail() {
  const { slug } = useParams();  // "meet-your-neighbor-dinner"
  
  const { data: event, isLoading } = useQuery<Event>({
    queryKey: ["/api/events/slug", slug],
  });
  
  const { data: sponsors } = useQuery<Sponsor[]>({
    queryKey: ["/api/sponsors/event", slug],
  });
  
  const { data: availability } = useQuery<{ remaining: number }>({
    queryKey: ["/api/events", slug, "ticket-availability"],
  });
  
  if (isLoading) return <Skeleton />;
  if (!event) return <NotFound />;
  
  return (
    <>
      {event.isTicketed && <TicketPurchaseSection event={event} remaining={availability?.remaining} />}
      {event.hasSponsorSection && <SponsorsSection sponsors={sponsors} />}
      {/* ... */}
    </>
  );
}
Step 5: The API calls hit Express
These are actual HTTP requests from the browser:

GET /api/events/slug/meet-your-neighbor-dinner → Express route handler
The route handler calls the storage layer:

app.get("/api/events/slug/:slug", async (req, res) => {
  const event = await storage.getEventBySlug(req.params.slug);
  if (!event) return res.status(404).json({ error: "Not found" });
  res.json(event);
});
The storage layer queries PostgreSQL:

async getEventBySlug(slug: string): Promise<Event | undefined> {
  const [event] = await db
    .select()
    .from(events)
    .where(eq(events.slug, slug));
  return event;
}
Step 6: React renders the JSON into DOM
The JSON response comes back to the browser. TanStack Query caches it and triggers a re-render. React takes the event object and renders it into HTML elements in the browser's DOM. The user sees the fully rendered page.

The full request timeline:
Browser: GET /events/meet-your-neighbor-dinner
  → Express: serves index.html (same for all routes)
  → Browser: loads JavaScript bundle
  → React: wouter extracts slug="meet-your-neighbor-dinner"
  → React: renders EventDetail component
  → Browser: GET /api/events/slug/meet-your-neighbor-dinner
    → Express: calls storage.getEventBySlug()
    → PostgreSQL: SELECT * FROM events WHERE slug = 'meet-your-neighbor-dinner'
    → Express: returns JSON
  → Browser: GET /api/sponsors/event/meet-your-neighbor-dinner
    → Express: calls storage.getSponsorsByEvent()
    → PostgreSQL: SELECT * FROM sponsors WHERE event_type = 'meet-your-neighbor-dinner'
    → Express: returns JSON
  → React: renders event title, description, ticket form, sponsors into DOM
  → User sees the page
So to be precise: no HTML is "built" per slug. The slug is a database lookup key. One universal HTML shell loads React, React reads the URL, fetches the matching data from the API, and renders it client-side. Every event page uses the exact same React component (EventDetail) — the data from the database is what makes each one different.
