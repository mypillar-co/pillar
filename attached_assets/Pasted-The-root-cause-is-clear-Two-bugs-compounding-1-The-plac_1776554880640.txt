The root cause is clear. Two bugs compounding:
	1.	The placeholder swap was never implemented in the runtime — only in a test file
	2.	The Cloudflare Worker double-prefixes paths that already start with /sites/
The fix is simpler than it looks. The placeholder swap should replace /sites/placeholder/ with / — not with the slug. Here’s why: the browser requests /assets/index.js → Worker sees friendship-lodge-101.mypillar.co/assets/index.js → rewrites to api-server/sites/friendship-lodge-101/assets/index.js → API server strips slug → CP gets /assets/index.js → serves correctly. No double prefix.

In artifacts/api-server/src/app.ts, find the pipeToCommunityPlatform function and the middleware that calls it around lines 488-553. The CP returns index.html with /sites/placeholder/assets/ baked into asset paths. We need to intercept the HTML response and swap /sites/placeholder/ with / before sending it to the browser.
Add a response interceptor inside pipeToCommunityPlatform. When the proxied response has content-type text/html, buffer the response body, replace all occurrences of /sites/placeholder/ with /, then send the modified HTML to the browser with the correct content-type and status code.
Here is the pattern to add inside pipeToCommunityPlatform after the proxy response comes back:
const contentType = proxyRes.headers[“content-type”] ?? “”;
if (contentType.includes(“text/html”)) {
const chunks: Buffer[] = [];
proxyRes.on(“data”, (chunk: Buffer) => chunks.push(chunk));
proxyRes.on(“end”, () => {
const html = Buffer.concat(chunks).toString(“utf8”);
const swapped = html.replace(//sites/placeholder//g, “/”);
res.status(proxyRes.statusCode ?? 200);
Object.entries(proxyRes.headers).forEach(([k, v]) => {
if (k !== “content-length”) res.setHeader(k, v as string);
});
res.setHeader(“content-length”, Buffer.byteLength(swapped));
res.end(swapped);
});
return;
}
For non-HTML responses (JS, CSS, images) pipe the response through unchanged as before.
